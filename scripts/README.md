# أدوات الأمان قبل النشر — نظام الأمراء

هذا المجلد يحتوي على أدوات دخلت الخدمة بعد اكتشاف عدة مخاطر حقيقية في عملية النشر
(اعتماد كامل على تعديل ملفات minified JS مباشرة، بدون staging، بدون اختبار تلقائي).

## القاعدة الإلزامية من الآن فصاعدًا

**قبل أي `git push` على هذا المشروع:**

```bash
bash scripts/check_before_push.sh
```

لو رجع بأي خطأ (`❌`)، **يُمنع النشر** حتى يتم حل المشكلة.

## اختبار تعديل قبل نشره (شبه-staging)

بما أن GitHub Pages لهذا المشروع في وضع "Legacy" (تنشر مباشرة من فرع `main`
بدون build step، وبدون دعم فروع متعددة)، الحل العملي البسيط هو:

1. عدّل الملف `.js` الجديد باسم منفصل تمامًا (مثال: `page-TESTNEW.js`)
2. شغّل:
   ```bash
   python3 scripts/make_preview.py <page>.html "<old_chunk_path>" "<new_chunk_path>"
   ```
3. افتح النسخة `preview-<page>.html` **محليًا فقط** (مثلاً عبر معاينة الملف)
   للتأكد بصريًا أن كل شيء يعمل قبل ربطها بالصفحة الحقيقية.
4. **لا تنشر أبدًا** أي ملف `preview-*` — هذه الملفات محمية بـ `.gitignore`
   بالإضافة إلى فحص إلزامي في `check_before_push.sh`.
5. بعد التأكد، استبدل الاسم في الصفحة الحقيقية (`<page>.html` و `<page>.txt`)
   بنفس الطريقة المعتادة، واحذف الملف التجريبي.

## لماذا هذا القيد موجود

- الـ repo هو مخرجات Next.js static export نفسها، بدون source TSX أصلي محفوظ.
- كل تعديل يتم مباشرة على كود مُصغَّر (minified) بأسماء متغيرات قصيرة متكررة
  عبر scopes مختلفة — هذا رفع فرصة الأخطاء الصامتة (مثل تضارب أسماء متغيرات،
  أو ربط زرار بدالة غير موجودة في الـ props) في أكثر من جلسة سابقة.
- GitHub Pages بهذا الوضع لا يوفر أي طبقة حماية تلقائية (لا build fail، لا تحقق).

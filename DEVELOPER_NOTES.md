# ملاحظات المطوّر — نظام الأمراء

هذا الملف **ليس** جزءًا من واجهة النظام أو بيانات المتجر. هو سجل داخلي يوثّق
القرارات المعلّقة والمخاطر المعروفة، بحيث لا تضيع بين الجلسات المختلفة حتى لو
تغيّرت الذاكرة أو انقطع السياق.

## 🔴 قرارات معلّقة — تحتاج إجراءً فعليًا

### 1. تدوير GitHub PAT المكشوف
**الأولوية: قصوى، لكنها مؤجّلة بطلب صريح من المستخدم حاليًا.**

يوجد GitHub PAT مكشوف في تاريخ محادثات سابقة مع هذا المشروع (لا يُكتب هنا
صراحةً — GitHub push protection يحجب أي commit يحتوي على نص الـ token الفعلي،
وهذا صحيح ومقصود). لم يُدوَّر حتى تاريخ هذا التحديث. يتطلب إجراءً مباشرًا من
المستخدم على github.com (Settings → Developer settings → Personal access
tokens) لإنشاء token جديد وإلغاء القديم، ثم تحديث أي مكان يستخدمه (لا يوجد
حاليًا استخدام معروف له خارج جلسات هذا المساعد نفسه).

### 2. `pos_fn_items_by_date` — دالة backend جاهزة، غير مربوطة بأي واجهة
ترجع قطع مخزون منتج معيّن مُرتّبة بتاريخ الاستلام. مفيدة محتملة لصفحة
`stocktake` أو `inventory` (سيناريو "متى استلمنا هذه القطع؟"). لم تُربط بعد
لعدم وجود طلب صريح — ليست عاجلة.

## ✅ قرارات مُنفَّذة (سجل تاريخي)

- **تحذير هامش الربح في POS** (`pos_fn_margin_warning`): كانت مبنية بالكامل في
  الـ backend، غير مربوطة بالفرونت اند. تم ربطها بحقل الخصم في نقطة البيع —
  للـ owner فقط (الدالة ترجع boolean فقط، لا تُسرّب رقم التكلفة).
- **تنظيف ملفات chunks ميتة**: تم اكتشاف وحذف عدة دفعات من ملفات `.js` غير
  مستخدمة عبر جلسات متعددة (`labels`, `customers`, `pos`, `products`) — راجع
  `scripts/audit_live_site.sh` للفحص الدوري.
- **أرشيف الكود المصدري**: أُنشئ `source/` لحفظ JSX الأصلي لأي مكوّن نبنيه
  نحن (بخلاف صفحات المطوّر الأصلي التي لا يوجد لها مصدر — راجع `source/README.md`).

## ⚠️ مخاطر بنيوية معروفة (لا حل كامل متاح حاليًا)

### GitHub Pages في وضع "Legacy"
ينشر مباشرة من فرع `main` بدون build step وبدون دعم فروع/staging متعدد.
الحل العملي البديل: `scripts/make_preview.py` (معاينة محلية قبل الربط).

### لا يوجد رصد أخطاء تلقائي (error monitoring) لواجهة العميل
لو حصل خطأ JavaScript فعلي عند العميل، لا توجد طريقة نعرف بها إلا لو أخبرنا
هو بنفسه. حل حقيقي (Sentry أو مشابه) يتطلب حساب خارجي — قد يخالف جزئيًا قيد
"صفر تكلفة" حسب الخطة المجانية المتاحة، يحتاج تقييمًا منفصلاً قبل التنفيذ.

### لا يوجد اختبار تلقائي (test suite / CI) بالمعنى الكامل
`scripts/audit_live_site.sh` بديل عملي جزئي (فحص بنيوي بعد النشر)، لكنه لا
يغطي المنطق الوظيفي (هل الزرار بيعمل الحاجة الصحيحة فعليًا) — فقط "هل الملفات
سليمة ومربوطة".
